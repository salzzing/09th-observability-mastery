## 3. 중요하다고 생각한 개념

### 개념 1. 상관관계

### 개념 설명

- 메트릭, 로그, 추적은 각각 독립된 신호이지만 이들을 서로 연결할 수 있어야 진정한 관측 가능성이 된다.
- 예를 들어 메트릭 히스토그램에서 이상 지점을 발견하면 → 이그젬플러를 통해 해당 추적으로 이동하고 → 스팬에서 원인을 분석한 뒤 → 로그로 상세 디버깅하는 흐름
- 로그에 trace_id를 넣거나, 메트릭에 이그젬플러를 붙이는 식으로 신호 간 이동 경로를 만든다.

### 중요하다고 생각한 이유

- 메트릭, 로그, 추적을 각각 따로 수집하는 것은 많은 조직에서 이미 하고 있지만, 장애 상황에서 이 세 가지를 빠르게 오가면서 원인을 좁혀나갈 수 없다면 Observability의 가치가 반감된다.
- 상관관계가 없으면 결국 데이터는 많은데 원인은 모르는 상황이 반복된다.

### 운영 관점에서의 의미

- 실제 장애 대응 시, 그라파나 대시보드에서 지연 스파이크를 발견했을 때 이그젬플러 클릭 한 번으로 템포의 추적 화면으로 이동하고, 거기서 문제 스팬의 로그를 로키에서 바로 조회할 수 있다면 MTTR(평균 복구 시간)이 크게 단축된다.
- 상관관계가 없으면 각 도구를 별도로 열어 시간대와 ID를 수동으로 맞춰야 하므로 장애 대응 속도가 느려진다.

---

### 개념 2. 구글 골든 시그널 4가지 (지연, 에러, 트래픽, 포화)

### 개념 설명

- 어떤 서비스를 모니터링할 때 "무엇을 봐야 하는가"에 대한 답이다.
- 지연(Latency)은 요청 처리 시간을 의미
- 에러(Error)는 실패 비율을 의미
- 트래픽(Traffic)은 시스템에 들어오는 요청량을 의미
- 포화(Saturation)는 리소스가 얼마나 가득 찼는지를 의미
- 이 네 가지를 기준으로 모니터링 항목을 설계하면 핵심을 빠뜨리지 않을 수 있다.

### 중요하다고 생각한 이유

- 모니터링 대상을 정할 때 체계 없이 "CPU랑 메모리 보자" 수준에 머무르기 쉬운데, 골든 시그널은 서비스 관점에서 무엇을 측정해야 하는지에 대한 명확한 프레임워크를 제공한다. 면접에서도 "어떤 항목을 모니터링할 것인가"라는 질문이 나왔는데, 이 프레임워크를 알고 있었다면 더 체계적으로 답할 수 있었을 것이다.

### 운영 관점에서의 의미

- 인프라 메트릭(CPU, 메모리, 디스크)만 보는 것은 포화(Saturation) 하나에만 집중하는 것
- 실제 사용자가 체감하는 문제는 지연이나 에러에서 먼저 드러나는 경우가 많다. 골든 시그널을 기준으로 대시보드를 구성하면 "서버는 괜찮은데 사용자는 느리다고 하는" 상황을 놓치지 않을 수 있다.

### 개념 3. 메트릭 유형 — 히스토그램과 이그젬플러(Exemplar)

### 개념 설명

- 메트릭은 카운터, 게이지, 요약, 히스토그램 네 가지 유형이 있음
- `히스토그램`은 응답 시간 같은 값을 버킷 단위로 분류해 분포를 보여주며, `histogram_quantile` 함수로 p95, p99 같은 백분위수를 계산할 수 있다.
- `이그젬플러`는 히스토그램의 특정 데이터 포인트에 추적 ID를 붙여서, 메트릭에서 추적으로 바로 이동할 수 있게 해주는 연결 장치이다.

### 중요하다고 생각한 이유

- 평균 응답 시간만 보면 대부분의 요청이 정상인 것처럼 보이지만, 실제로 일부 사용자는 심각한 지연을 겪고 있을 수 있다.
- 히스토그램과 백분위수를 통해 꼬리 지연(tail latency)을 포착할 수 있음
- 이그젬플러를 통해 그 느린 요청이 구체적으로 어떤 경로를 탔는지까지 추적할 수 있음

### 운영 관점에서의 의미

- SLO를 "p99 응답시간 500ms 이하"처럼 설정할 때 히스토그램이 반드시 필요하다. 또한 그라파나에서 히스토그램 위의 이그젬플러 점을 클릭해서 바로 추적 화면으로 넘어가는 워크플로우는, 장애 원인 파악 시간을 분 단위에서 초 단위로 줄여줄 수 있다.

### 4. 이해가 어려웠던 부분

### 어려웠던 부분 : 스팬

- 스팬은 분산 추적에서 하나의 작업 단위
- 사용자 요청이 여러 서비스를 거칠 때 각 단계가 하나의 스팬
- 스팬끼리 부모-자식 관계로 연결되어 전체 요청 흐름과 병목 지점을 트리 형태로 파악 가능

> **AI의 정리**
> 
> 
> 스팬은 분산 추적에서 하나의 작업 단위입니다. 사용자가 주문 API를 호출하면 API 서버 → 인증 서비스 → DB 조회 → 결제 서비스 이런 식으로 여러 단계를 거치는데, 이 각각이 하나의 스팬입니다.
> 
> 스팬 안에는 작업 이름, 시작 시간, 소요 시간 같은 기본 정보가 들어가고, 태그로 `http.method=GET` 같은 메타데이터를 붙일 수 있습니다. 스팬 콘텍스트에는 추적 ID와 스팬 ID가 담겨 있어서, 서비스 간 호출 시 HTTP 헤더를 통해 전달되면서 "이 요청이 어느 추적에 속하는지"를 알려줍니다.
> 
> 스팬끼리는 부모-자식 관계로 연결됩니다. 주문 API 스팬이 부모, 그 안에서 호출한 결제 서비스 스팬이 자식(child of)입니다. 이 관계를 따라가면 하나의 요청이 어떤 서비스를 어떤 순서로 거쳤는지 트리 형태로 보이고, 어디서 병목이 생겼는지도 바로 파악됩니다.
>
